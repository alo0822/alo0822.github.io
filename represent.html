Represent!

March 2016

Android Development, UI Design

Represent! is an Android mobile and wear application that was built for an assignment for the User Interface and Development class (CS160).

OVERVIEW

What: An Android mobile and wear application that provides delivers facts on congressional representatives

Goal: Provide easily accessible information on congressional representatives

Users: Voters on the go

Built with: Android, Google Location, Google Geocoding, Twitter, Sunlight Foundation API

See the demo video for Represent! here.

DESIGN

For this application, I wanted to create a clean and straightforward user interface so users can receive information quickly and easily. For this assignment, we were given the following view requirements:

  • Main View (Mobile): A home screen welcoming users and allowing them to enter a zip code or use their current location.
  • Congressional View (Mobile): Display each member of Congress based on location with full name, photo, party, email, website, and latest tweet
  • Detailed View (Mobile): Detailed view of a member with full name, photo, party, end date of term, committees assigned, and bills sponsored
  • Main View (Watch): A shortcut glance on the watch that loads the list of representatives looked up from the phone
  • 2012 Vote View (Watch): 2012 US Presidential vote for the same county as the House Rep with the state, county, vote % for Obama, and vote % for Romney.
  • Shake (Mobile): When the user shakes the watch, it selects a random location in the United States and updates the congressional representatives information accordingly.

In the first phase, I generated wireframes of each screen with Figma to quickly lay out how I envisioned each view. I decided a serif typeface for the title as it fits the more official feel of elections. The color scheme would be the American colors of red, blue, and white. For the "Represent!" title, I some research for inspiration and came across the classic VOTE button with two lines of white stars. From there, I decided on a more minimal design a thin red line and a single row of white stars below the title.

Wireframes for mobile and wear view

DEVELOPMENT

In the second phase, I coded the wireframes I designed into functional Android code, and I implemented the functionality of phone and watch interaction. For this phase, I manually inputted data to show in the rendered screens. However, I set up the backend and data structures in preparation for use with APIs.

In the third phase, I added code to properly interface with APIs that provide real live data. We used the following APIs:

  • Google Location: Used to get the phone's current location
  • Google Geocoding: Used to get the county from the phone's current location or zip code to retrieve 2012 Presidential Vote data
  • Twitter: Used to get the representative's last tweet and their photo
  • Sunlight Foundation API: Used to get all congressional information about the representatives including party, committee assignments, and bills sponsored
  • 2012 Voting Data in JSON: Used to get data for 2012 Presidential Vote

FINAL PRODUCT

Final design for mobile views
Final design for watch views

There were a few design changes I made in the final product from the first design phase. The most significant ones were:

  • Implementation of Tweet View: In my first design, there was a “Latest Tweet” title and text showing the representative’s latest Tweet. After incorporating the Twitter API, I was able to add a TweetView instead. With a TweetView, users can easily view the Tweet in Twitter by clicking on the Twitter icon that was embedded. Since I used a TweetView instead of plain text, I made the “Latest Tweet” title smaller to balance out the larger text in the TweetView.
  • Details View text and spacing: I increased the line spacing between and within both committees and bills. There is a lot of information and text present in this view, so this greatly increases the readability of the page. The text color of the introduced dates is also lighter to make the bills section more readable. 
  • Removing watch notifications: Originally, loading the list on mobile will send a notification to the watch, prompting the user to tap the screen and view the list of representatives. Instead, I implemented the phone-to-watch communication so that loading the list on the phone will automatically load the list on the watch. This way, the users don’t have to do one extra step to tap on the screen in order to view the data.
  • Tapping on rep on watch to view Detailed View: In the initial design for the watch application, when the user taps on a representative a screen will load to prompt the user to tap again to view the detailed view of that representative on the phone. Similar to the last change, I removed this screen to simplify the user flow. The extra “confirmation” tap was unnecessary in this case, since tapping on the representative doesn’t change the current watch view in any way and is an almost instantaneous load on the phone.
  • Removing photos in watch Main View: In the original design, I had an image of the representative, and then the representative’s name and party. However, while I was implementing the APIs, I had trouble sending images from the phone to the watch. I was short on time, so I decided to omit the image and place the representatives name and party in the center of the screen to increase readability.
