musicDB

November 2015

Database Management, Web Development, UI Design

musicDB is a web application with searchable music metadata that was the final project for the Database Management class (INFO257). With the Spotify and SetlistFM APIs, we collected a sizeable amount of information, including 44 individuals, 40 groups, 769 releases, 965 concerts, and 17,925 songs.

OVERVIEW

What: A web application with searchable music metadata

Goal: Create a space where information about music is consolidated and easily accessible

Users: Music enthusiasts and artists

Built with: Ruby on Rails, HTML&CSS, Google MDL, JavaScript

My Role: Database design, UI design, front end development

DATABASE DESIGN

In order to accomplish our goal, the database needed to have a simple yet accurate depiction of the music industry. We focused on how to successfully represent the music aspect of the database, as opposed to social aspects that are commonly associated with listening to music. The database breaks the concept of the music industry to seven separate entities:

  • Individuals: Individual people that make up the music industry and have attributes such as name, birthday, primary and secondary instruments, etc.
  • Members: An intermediate entity that bridges the connection between individuals and groups, and includes information related to that relationship
  • Groups: Bands that release music and can consist of one or many individuals.
  • Releases: Different types of media that can be created by a group
  • Songs: Individually titled pieces of music that groups perform
  • Tracks: Utilized by Songs in order to provide additional data about how the song connects to Releases and Shows
  • Shows: Concerts that groups perform
  • Record Labels: Entities that distribute the releases

These entities are the simplest depictions of the many parts that make up the record industry that allow for an accurate and easy way to interpret representation of the metadata related to music.

Entity-relationship diagram for musicDB

While this database is a very fair representation of the music industry, there are a couple design decisions that could be improved upon:

  • Include a 1:N relationship between artists and songs: This would allow us to find the original performing artist of a particular song in an easier manner, and for cover performances to be much better represented.
  • Record Label should connect to the Individual entity: In our design, the individuals that are stored are stored as strings, as opposed to having a connection to the Individual entity, allowing for the storage of more information about those people.
  • Incude social aspects of the music discovery: We could create a User entity for the users of the database, a Comments entity that would allow for users to comment on the the pages of other entities, allowing for interaction with other users.
  • Create a separate entity for Genre: This would allow for better interconnection between separate songs, releases, and groups, as the process of joining tables on this category would be vastly improved.

WEBSITE DESIGN

For the UI interface, I decided to design a website that is simple, elegant, and intuitive to use. Since the main purpose of our database is searching for artists and music, the homepage serves as the main simple search web page with a link to the advanced search.

Home page for musicDB

Clicking on the Search tab in the navigation bar also brings the user to a quick search page.

Quick Search page

The Advanced Search page has multiple search bars for complex queries and includes three drop down menus for each search bar. The first menu is for comparators (=, <, >), the second menu lists the possible attributes for each entity, and the third menu includes conditionals (and, and not, or). The drop down menus allow the user to make complex queries.

In order to allow the user to make queries with more than one entity or attribute, we had to include multiple search bars. However, we had some difficulties with JavaScript and were short on time, so we were unable to implement the Advanced Search page exactly as originally designed. This is very excessive and an overall poor design. Our original design was to set the default as two search bars, with the option to add or remove individual searches.

Original design for Advanced Search page

In order to show results from each entity (Group, Individual, Song, etc.), we used a tabs, where each tab represents an entity.

The information page for each result displays all the attributes that the database has information for.

The input data form allows users to input information for each type of entity.

Input form, with the Individual form selected.

Overall, the UI provides basic inputs that allow users to populate and add seed data while being able to fully make queries. With more time, we would make improvements to the user interface so that it would more accurately reflect the original design. However, this class was not focused on UI design so we did not focus on the design aspects of the project.
