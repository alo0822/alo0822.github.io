SafeSpace

March - April 2016

Web Development, UI Design

SafeSpace is a web application that I developed with a team for a project customer as part of the Software Engineering class (CS169).

OVERVIEW

What: An online anonymous messaging application

Goals:

  • Improve the mental well-being of undergraduate students at UC Berkeley
  • Provide a safe space for peer support and student interactions

Users: All undergraduate students at UC Berkeley, regardless of class, race, or background

My Role: UI design, front-end development, back-end development

Built with: Ruby on Rails, HTML&CSS, Bootstrap

ITERATION 0

The goal of our first meeting with the customer was to understand the goals and scope of the project. The problem the customers wanted to address was the lack of mental health resources available to students at UC Berkeley and the mental health stigma. The customers wanted to create an application that was college friendly, anonymous, and easy to use.

We identified three main features of the app:

  • An instructional video and quiz in the sign up process for all users
  • Filter users through ethnicity, age, mental health, etc.
  • Add and chat with buddies

During this meeting, we also drew up some rough sketches with the customer. Since the messaginig component was the main feature, we decided that it should be the first page for a logged in user.

Rough sketches of landing page, search, messaging, and profile settings

ITERATION 1

For this iteration, I created initial wireframes based on the sketches we came up with from last meeting.

Initial wireframes of landing page, search, and messaging

From here, there were a few changes we made before starting on a higher-fidelity prototype for next iteration. There were two significant features that we added:

  • Peer Counselors are trained student counselors with their own general accounts to assist in more difficult situations. Users can recommend their buddy to a peer counselor, and peer counselors don't have to add a user to talk to them.
  • The emergency protocol button was moved a more visible and accessible location so users can quickly look for protocol instructions in the event of a crisis.

For the back end, we decided to first start by creating Users and implementing sign up and log in features.

ITERATION 2

After making the changes discussed from the last meeting, these were the higher-fidelity mockups.

Higher-fidelity mockups for landing page, search, and messaging

For the back end, we implemented a basic messaging system (sending and receiving messages) and user status. A user will be shown as offline after a certain amount of time of inactivity. In this phase, a user can see and chat with all the users that exist.

ITERATION 3

During the third iteration, we focused on implementing web sockets with Faye for real-time messaging. We also added notifications, so users receive a notification when they receive a new message, and appearing offline, so users can choose to appear offline to other users.

ITERATION 4

In this iteration, we implemented friend requests, search and filter, and setting user profile characteristics.

With friend requests, the user needs to send another user a friend request that needs to be accepted in order for them to start messaging. Users can add and remove friends, and accept or delete friend requests.

Search and filter allows users to filter through all the users based on different filter options. These options were defined by the customers, and were based on mental health disorder, gender, ethnicity, academic focus, and religion.

We also implemented the ability to set user profile characteristics. These profile characteristics are what the search and filter functionality uses to filter through users. This gives the users the ability to set and change their profile characteristics.

FINAL WEBSITE AND POSTER

REFLECTION

During this project, we focused mainly on implementing the backend functionalities for the Users, messaging, and search and filter. We were able to finish most of the important features that the customers wanted. Some of the features we had discussed but didn't implement are:

  • Smaller, more specific features the customers wanted.
  • The sign up process with an instructional video and a quiz. The details for this from the customers were not ready before our project was due.
  • User interface designs. We focused more on completing the backend functionalities, so we didn't have enough time to finalize the user interface and apply the lastest designs.

From this project, I learned:

  • Web development with Ruby on Rails.
  • How to work with customers to create a product based on their requirements. We had weekly meetings with our customers to keep them updated about the progress of the project, check in with them for any changes in their requirements, and discuss the details of the features we were implementing next.
  • To work more efficiently with a team. We met up at least once a week to sync up on the tasks that we were working on. Communication was very important, especially since we were not able to meet up everyday.
  • To plan for a larger project. This was the largest project that I had taken on before, and we had to plan everything ourselves from the beginning. We discussed with the customers on how they wanted to prioritize the features, and created a timeline for the team of when to work on which parts. We also used Pivotal Tracker to keep track of our stories and tasks.

Although we didn't get to some of the tasks that we had wanted to, we still completed a significant amount of work and had created a working product that included all the basic functionalities. Users can sign up, find others, add friends, and message in real-time. Our customers were very happy with what we were able to accomplish in the time we worked together, and it was a great experience for all of us.
